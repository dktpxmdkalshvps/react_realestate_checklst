## 🎯 프로젝트 목적

### 배경

한국에서 부동산을 매수하거나 노후 주택을 수리하려는 실수요자는 다음과 같은 어려움에 직면합니다.

- **지자체마다 다른 지원사업**: 서울·경기·인천·부산·대구 등 지역별로 지원 금액, 자격 요건, 신청 기간이 모두 달라 한 곳에 정리된 정보를 찾기 어렵습니다.
- **복잡한 구비서류**: 사업 유형(집수리·청년 주거·농어촌 정비)에 따라 요구 서류가 다르고, 발급 시기 제한(공고일 이후 발급분)까지 고려해야 합니다.
- **임장 시 놓치기 쉬운 항목**: 등기부등본 권리분석, 불법 증축 여부, 점유 확인 등 전문적인 확인 사항이 많습니다.
- **2026년 제도 변화**: 스트레스 DSR 3단계, 다주택자 양도세 중과 배제 종료(5월 9일), 자금조달계획서 가상자산 기재 의무화 등 새로운 규제가 다수 시행됩니다.

### 해결 방향

세 가지 핵심 기능을 하나의 웹 앱으로 통합했습니다.

```
지원금 정보 조회  ──►  구비서류 안내  ──►  임장 체크리스트
(어디서 얼마?)       (뭘 준비하지?)       (현장에서 확인할 것)
```

- 별도 회원가입·로그인 없이 바로 사용 가능한 **정적 웹 앱**
- Vercel을 통한 **무료 배포** 및 즉시 접근
- 전문 지식 없는 일반인도 이해할 수 있는 **카드형 UI**

### Tailwind CSS를 선택한 이유

```
일반 CSS       — 클래스명 충돌, 스코프 관리 번거로움
CSS Modules   — 설정 복잡, 동적 색상 적용 어려움
Styled Comp.  — 런타임 오버헤드, 번들 크기 증가
Tailwind CSS  ✔ 유틸리티 클래스로 즉시 스타일링, 반응형 손쉽게 처리
```

## 🔑 핵심 기능

### 탭 1 — 지역별 집수리 지원금

- **7개 지역** 버튼 선택: 서울 / 경기 / 인천 / 부산 / 대구 / 세종대전 / 농어촌
- 선택한 지역의 지원 프로그램이 **색상 구분 카드**로 즉시 렌더링
- 카드별 대상 주택 · 신청 자격 · 지원 금액 · 공사 범위 · 주의사항 · 신청처 표시
- **사이트 바로가기** 버튼 → 해당 기관 홈페이지 새 탭 오픈
- 하단 **전국 공통 지원사업** 3종 고정 표시 (주거급여 수선유지급여 / 그린리모델링 / 슬레이트 철거)
- **위반건축물 기준** 3단계 상세 안내 (원칙 제외 / 양성화 옥탑방 예외 / 반지하 요건)

|------|--------|------------|

| 서울 | 안심 집수리 융자 | 6,000만원 (연 0.7%) |
| 경기 | 소규모 노후주택 집수리 | 1,600만원 (공용부) |

| 인천 중구 | 저층주거지 재생 | 1,200만원 |
| 부산 | 희망의 집수리 | 18개 항목 현물 지원 |
| 대구 북구 | 공용시설 수리비 | 공사비 70% |
| 세종·대전 | 주거취약가구 수리 | 400만원 |
| 농어촌 | 주택개량 저금리 융자 | 2억5천만원 |
| 농어촌 | 빈집 철거 보조 | 400만원 |
| 전국 | 슬레이트 지붕 철거 | 700만원 |

### 탭 3 — 임장 체크리스트

- **6개 섹션 37개 항목** 체크박스
- **실시간 진행률 바** (0~60%: 주황 / 60~99%: 파랑 / 100%: 초록)
- **필수(★) 항목** 미완료 개수 실시간 배지 표시
- **섹션별 완료 카운트** 헤더에 실시간 업데이트 (`3/6`)
- 체크 완료 항목 취소선 + 회색 처리
- 매물 유형 선택 (주택 / 토지 / 상가·오피스텔) 시 추가 주의사항 표시
- **전체 초기화** 버튼
- 하단 **계약 전 핵심 6단계** 순서 안내

## 🏗 데이터 구조 설계

### regionData 스키마

```js
const regionData = {
  "지역명": {
    programs: [
      {
        name:   string,   // 사업명
        target: string,   // 대상 주택
        who:    string,   // 신청 자격 (\n 줄바꿈 허용)
        amount: string,   // 지원 금액
        works:  string,   // 지원 공사 범위
        note:   string,   // 주의사항·기간
        apply:  string,   // 신청처
        url:    string,   // 사이트 URL
        color:  string,   // Tailwind 색상 키
      }
    ],
    contact: string,      // 대표 문의처
    tip:     string,      // 활용 팁
  }
}
```

### docsData 스키마

```js
const docsData = [
  {
    category: string,
    subtitle: string,
    icon:     string,   // 이모지
    color:    string,
    groups: [
      {
        title: string,
        items: [
          { text: string, required: boolean }
        ]
      }
    ],
    tip:      string,
    warnings: [          // 빈 배열 가능
      { title: string, body: string }
    ]
  }
]
```

### checklistData 스키마

```js
const checklistData = {
  "섹션명": {
    icon:  string,
    color: string,
    items: [
      {
        id:       string,   // 고유 ID (d1, r1, b1...)
        text:     string,
        critical: boolean   // true → 필수 뱃지
      }
    ]
  }
}
```

> **ID 네이밍 규칙**: `d`=기본서류, `r`=권리관계, `b`=건물상태, `e`=주변환경, `o`=점유확인, `p`=가격협상

## 🔧 개발 과정 및 이슈

### 개발 순서

```
1단계  데이터 수집·검증 (지자체 공고문, 정부24)
   ↓

   ↓
3단계  UI 컴포넌트 구현 (탭 구조 → 지원금 카드 → 서류 → 체크리스트)
   ↓
4단계  3차례 데이터 업데이트 (오류 수정 + 위반건축물 기준 상세화)
   ↓
5단계  Vercel 배포 후 레이아웃 버그 수정
```

### 이슈 1 — 데이터 정확도 오류 (3차례 수정)

**문제**: 초기 버전에서 지자체 공고 내용과 다른 정보가 발견됐습니다.

| 항목 | 초기 (잘못된 정보) | 최종 (정확한 정보) |
|------|-------------------|-------------------|
| 대구 북구 세대 기준 | "8세대 이상" | **20세대 미만** 소규모 공동주택 |
| 슬레이트 지원금 | "철거비 전액" | 최대 **700만원** |
| 경기 빌라 연한 | "20년 이상" | 단독 20년 / 빌라 **15년** 이상 |
| 세종대전 지원 항목 | "도배·장판" | **창호·단열·난방** |
| 서울 신청 기간 | "4월경" | 보조: 4월 / 융자: **상시** |

**해결**: 데이터를 컴포넌트와 완전히 분리된 `const` 객체로 관리해, UI 코드를 건드리지 않고 데이터만 교체하는 구조로 설계.

### 이슈 2 — 위반건축물 기준 불명확

**문제**: 초기 버전에서 "불법 건축물 제외" 한 줄로만 처리했으나, 실제로는 3가지 세부 기준이 존재합니다.

**추가된 기준**:
```
① 건축물대장에 위반건축물로 표기된 주택 → 지원 불가 (원칙)
② 양성화된 옥탑방 → 건축물대장에 양성화 사실이 명확히 기재된 경우에 한해 예외 허용
③ 반지하 주택 → 건축물대장·현황이 실제 지하층이어야 하며, 창고 등 비주거 용도 사용 시 제외
```

**반영 위치**: 서울 `note` 필드 + 구비서류 탭 `warnings` 배열에 별도 경고 박스로 이중 표시.

### 이슈 3 — Vercel 배포 후 좌우 빈 공간 발생

**문제**: 로컬에서는 정상이었으나, Vercel 배포 환경에서 콘텐츠가 화면 중앙에 좁게 표시되고 양쪽에 빈 공간 발생.

**원인**: 각 탭 콘텐츠 최상위 div에 `max-w-2xl mx-auto`(최대 672px)가 설정되어 있어 데스크톱 넓은 화면에서 좌우 여백이 강제로 생성됨.

**해결 과정**:

```
1차 시도: max-w-2xl → max-w-5xl (1024px)  — 여전히 여백 존재
2차 시도: max-w 완전 제거 → 해결
```

```jsx
// Before
<div className="p-4 max-w-2xl mx-auto">

// After
<div className="p-4">
```

### 이슈 4 — 검은 배경 영역 발생 (오른쪽 빈 공간)

**문제**: 콘텐츠 오른쪽에 검은색 배경 영역이 노출됨.

**원인 1**: 루트 div에 너비 지정이 없어 앱이 콘텐츠 너비만큼만 렌더링되고, 나머지 영역은 Vite 기본 템플릿의 `body` 배경색(다크)이 노출됨.

**원인 2**: `w-screen` 적용 시 스크롤바 너비(약 17px)까지 포함되어 가로 오버플로우가 발생, 오히려 스크롤바가 생겨 더 어색해짐.

**해결**:

```jsx
// 1단계: w-screen 시도 → 가로 스크롤 오버플로우 유발, 실패
className="bg-gray-100 w-screen"

// 2단계: w-full + overflow-x-hidden + useEffect로 body 배경 강제 설정
className="bg-gray-100 w-full min-h-screen overflow-x-hidden"

useEffect(() => {
  document.body.style.background = "#f3f4f6";
  document.body.style.margin = "0";
  document.body.style.padding = "0";
  document.documentElement.style.background = "#f3f4f6";
}, []);
```

**핵심 포인트**: Vite 프로젝트의 `App.css`나 `index.css`가 body에 다크 배경을 적용하고 있으므로, React 컴포넌트 마운트 시 `useEffect`로 강제 덮어씌워야 함.

### 이슈 5 — 조건부 렌더링에서 탭 전환 시 체크 상태 유지

**문제**: `{tab === "checklist" && <div>...</div>}` 조건부 렌더링 방식에서 탭을 전환했다가 돌아오면 체크 상태가 초기화될 것을 우려.

**분석**: 체크 상태(`checks`)가 최상위 `App` 컴포넌트의 `useState`로 관리되므로, 탭 전환과 무관하게 상태가 유지됨. 탭 전환은 단순 조건부 렌더링이라 state가 `App`에 있는 한 유실되지 않음.

**결론**: 구조 변경 없이 그대로 유지.
